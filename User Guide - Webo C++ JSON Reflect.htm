I.    INTRODUCTION

 

A. Motivation

 

C++ JSON Reflect provides a reflection toolkit for C++ and a class schema representation based on the JavaScript Object Notation (JSON). The toolkit embeds a JSON-based type system into C++ programs that forms the basis for a meta-object protocol (MOP) to maniuplate objects and their classes at run-time. The libraries build on this type system to provide a JSON encoding for marshaling classes from/to an input or output stream. A ClassLoader is provided for loading classes from the file system, also allowing for loading classes from the network.

 

Benefits

 

·         C++ embedded type system and run-time reflection API

·         Object API for JSON parsing and generation (JSON.Stringify(), JSON.Parse())

·         Object API for generating XML (XML.Stringify())

·         JSON-based Class schema and ClassLoader

 

 

B. Packages and Name Spaces

 

The toolkit is packaged into four DLLs, layered as follows:

                       

      encoding.dll      encoding_xml.dll

               \       /

              modeling.dll

 

The following name spaces are defined by the toolkit and are structured with corresponding file system directories:

 

webo::modeling                - reflection API

webo::modeling::datatype      - datatypes defined by the toolkit

webo::encoding                - JSON and schema encoding and decoding

webo::encoding_xml            - XML encoding

 

 

C. Third Party Dependencies

 

The toolkit uses standard C++ and does not hold any dependencies other than the Boost libraries. Individual components may not have any third party dependencies at all. For example, the modeling library may be built without installing Boost:

 

            modelling.dll                 - no third party dependencies

            encoding.dll                  - boost header dependency for JSON parsing

            encoding_xml.dll           - boost header dependency for XML parsing

 

 

II.   DATA TYPES

 

A. Type and DataType

 

All types in the toolkit derive from webo::modeling::Type and have a name and nameSpace which defaults to the default name space.

 

All built-in data types in the toolkit derive from webo::modeling::datatype::Datatype, which itself is a direct descendent of Type.

 

User defined instances of webo::modeling::Class are also supported. Class is also a Type.

 

This is illustrated as follows:

 

                        Type

                          |

                          | <<is a>>

                          -------------------------------------------------------------

                          |                                                          |

                        DataType                                          Class

                          |                                                             \\

                          | <<is a>>                                                \\  <<instance of>>

                          ------------------------------------------     -------------------------

                          |                    |                 |                \\                     \\

                          |                    |                 |               UserClass1   UserClass2

                        tCharacter  tBoolean    ...

                            \\     

                             \\ << instance of>>

                          tCharacter::Instance()

 

 

Types are prefaced with the character 't' to prevent conflicts with symbols defined by the host langauge or its commonly used libraries. For example, String is likely to be a commonly defined symbol; so, we use tString as our type name.

 

 

B. Simple Data Types

 

The following simple data types are built-in: tCharacter, tBoolean, tInteger, tFloat, tString and tNull. These types correspond to the full set of JSON supported types, with a few deviations. tCharacter converts to/from JSON string to directly represent the C++ char type. tInteger and tFloat convert to/from JSON number to directly represent the C++ int and double types.

 

All simple data types come with two methods:

 

instance() - factory method for the corresponding singleton class

value(<value>) - creates an instance using the given value

 

For example, the following would declare and create an instance of tInteger:

 

            shared_ptr<Object> aNumber = tInteger::value(5);

 

And, the following assertion would hold true:

 

            assert( aNumber.type == tInteger::instance() )

 

 

C. Simple Arrays

 

The tArray<> templated data type supports simple base types, i.e., tArray<tCharacter>, tArray<tBoolean>, tArray<tString>, tArray<tInteger> or tArray<tNull>. The reflection API requires all objects to have a type, and arrays to have a fixed length and single base type; unlike JSON, arrays are homogeneous. Also, multi-dimensional arrays are not currently supported.

 

As with simple data types, tArray<> comes with an instance() method and various value() methods for creating reflection types using native C++ arrays:

 

instance() - factory method  for referencing the corresponding singleton class

value() - create an empty array

value(size_tlength) - create array with defaults values

value(char[], size_tlength) - create initialized character array

value(int[], size_tlength) - creates initialized integer array

value(string[], size_tlength) - creates initialized string array

value(bool[], size_tlength) - creates initialized boolean array

 

Default values are as follows : tCharacter(0), tInteger(0), tString(""), tBoolean(false).

 

For example, the following would declare and create an instances of tArray with base type tInteger:

 

       int integers[] = { 1, 2, 3 };

            shared_ptr<Object> arr1 = tArray<tInteger>::value(integers, 3);

      shared_ptr<Object> arr2 = tArray<tInteger>::value(1);

 

And, the following assertions would hold true:

 

            assert( arr1.type == tArray<tInteger>::instance() );

      assert( arr1[0] == tInteger::value(1) );

      assert( arr1[1] == tInteger::value(2) );

      assert( arr1[2] == tInteger::value(3) );

            assert( arr2.type == tArray<tInteger>::instance() );

      assert( arr2[0] == tInteger::value(0) ); // default value

 

Be aware that as with the C/C++ language, array access with out-of-bounds indexes are not checked. In particular, constructing an array that has a larger size than the native array argument may not result in an explicit error, but would result in corrupted data. Similarly, access to out-of-bounds indexes of an array object may not result in an explicit error, but would result in invalid return data.

 

The following provides an example of looping over a simply array:

 

std::shared_ptr<Object> *nativeArr = arr1->asArray<tInteger>();

 

for (size_t i=0; i<arr1->getArrayLength(); i++) {    

   assert( nativeArr[i]->type == tInteger::instance() );

      }

 

 

C. Object Arrays

 

The tObjectArray data type supports a base type of any user defined class. The reflection API requires all objects to have a type, and arrays to have a fixed length and single base type; unlike JSON, arrays are homogeneous. Also, multi-dimensional arrays are not currently supported.

 

tObjectArray<> comes with an instance() method and a three value() methods:

 

instance() - factory method  for referencing the corresponding singleton class

value(Class & baseType) - creates an empty array

value(Class & baseType, size_t length) - create array with defaults values

value(Class & baseType, std::shared_ptr<Object> arr[], size_t length) - create initialized array

 

For example, the following would declare and create an instance of tObjectArray with base type MyClass:

 

            Class c("MyClass");

            shared_ptr<Object> arr = tObjectArray::value(c, 2);

 

And, the following assertions would hold true:

 

            assert( arr.type == tObjectArray::instance(c) );

      assert( arr[1]->type == c );

 

Be aware that as with the C/C++ language, array access with out-of-bounds indexes are not checked. In particular, constructing an array that has a larger size than the native array argument may not result in an explicit error, but would result in corrupted data. Similarly, access to out-of-bounds indexes of an array object may not result in an explicit error, but would result in invalid return data.

 

The following provides an example of looping over an object array:

 

std::shared_ptr<Object> * nativeArr = arr->asObjectArray(c);

 

for (size_t i = 0; i < arr->getObjectArrayLength(); i++) { 

   assert( nativeArr[i]->type == c );

}

 

 

II.   USER DEFINED CLASSES

 

A. Class, Object and ClassModel

 

The toolkit allows you to define your own classes by instantiating the Class class.

 

The concepts define a meta-hierachy that is embedded in a host language, i.e. C++. This can be a point of confusion since the embedded meta-hierarchy is defined in terms of the host language meta-hierarchy. The correspondence is as follows:

 

·         The singleton metaclass, called "Class", is defined as a C++ Class.

·         User defined classes are defined as C++ objects that instantiate the "Class" class.

·         Instances of user defined classes are represented by instances of  the "Object" class.

 

The C++ Programming Language does not define a mechanism for dynamic (run-time) class creation. Within C++, an example meta-hierarchy correspondence would be as follows:

 

            metaclass "Class"         ------------>          C++ Class

            user class "MyClass"     ------------>          C++ instance of "Class"

            instance of "MyClass"   ------------>          C++ instance of "Object"

 

The toolkit provides a factory class called ClassModel, for creating and storing a context of defined classes. Although, classes do not have to be defined as part of a ClassModel, the class model context provides checks for unique class names. However, any duplicate named classes outside and across ClassModel instances would not be detected as a different class, and would pass the equality test even if they contained different attributes.

 

 

Object Equality

 

Three forms of equality checking are available for simple types, as shown in the following examples:

 

      assert( *tString::value("test1") == *tString::value("test1") );

      assert( tString::value("test1")->Equals(tString::value("test1")));

      assert( tString::value("test1")->AsString() == "test1" );

 

In contrast, equality checking of instances of user-defined classes compares by identity and does not perform a deep comparison of attributes.

 

 

B. Class Attribute

 

Any number of attributes may be added to a user defined class using the addAttribute() method which takes as parameters, a name for the attribute and a type. The type may be any built in data type or user defined class.

 

For example, the following would define a user class named MyClass with two attributes of the simple types tInteger and tCharacter:  

 

      Class c("MyClass");

      c.addAttribute("var1", tInteger::instance());

      c.addAttribute("var2", tArray<tCharacter>::instance());

 

And, the following would add another attribute with a type defined by a user defined class:

 

      // define a new class

      Class c2("MyClass2");

      c2.addAttribute("a", tString::instance());

 

      // store a new attribute with type of the new class

      c.addAttribute("var3", c2);

     

Here is how you would instantiate the object and set values on all its attributes:

 

      Object o(c);

      o.set("var1", tInteger::value(42));

      o.set("var2", tArray<tCharacter>::value(3));

      o.get("var3")->set("a", tCharacter::value('z'));

     

Attributes can be dynamically added to a class even after an object of that class has been instantiated. For example, the object o defined above would have a new accessible attribute even after having instatiated it:

 

      c.addAttribute("var4", tCharacter::instance());

      o.set("var4", tCharacter::value('z'));

 

 

C. Class Instantiation

 

Instance of User Defined Class

 

When an object is an instance of a user defined class, its attribute values may be accessed using the get() and set() methods. Object attribute values are stored as std::shared_ptr<Object>. When an attribute is added, it is initially null. On the first get() call, the attribute is instantiated, except for attributes of type tArray<> or tObjectArray since their desired size is not known. Elements of an array type attribute must be set before being accessed or their value will be null.

 

To check if an object's attribute has been initialized to a value, use the following method:

 

            bool hasValue(std::string name)

 

To check if an object is set to a tNull value, use the following method:

 

            bool isNull()

 

 

Instance of Built In Data Type

 

When an object is an instance of a built in data type, its value would be accessed using one of the following methods:

 

std::string asString()

char asCharacter()

int asInteger()

bool asBoolean()

 

Corresponding methods are used to store values of simple built-in types into an existing object:

 

asString(std::string)

asCharacter(char)

asInteger(int)

      asBoolean(bool)

 

For array types, the object must be converted to a native C/C++ array for access to its elements using the [] operator:

 

template <typename T> std::shared_ptr<Object> * asArray()

std::shared_ptr<Object> * asObjectArray(Class & baseType)

 

 

Instance of Simple Array

 

When an object is an instance of a simple array, its element values would be accessed by first converting the objecto a native C/C++ array and then using the [] operator. Array elements are initialized with default objects at creation time. The elements may be replaced with new objects if so desired.

 

For example, the following would get and set a value on an array object element and then replace an element of the array with a new object:

 

      std::shared_ptr<Object> arr= tArray<tBoolean>::value(2);

      std::shared_ptr<Object> myBool = arr->asArray<tBoolean>()[1];

      arr->asArray<tBoolean>()[0]->asBoolean(true);//replace value

      arr->asArray<tBoolean>()[0]=tBoolean::value(true);//replace object

 

 

Instance of Object Array

 

When an object is an instance of an object array, its element values would be accessed by first converting the object to a native C/C++ array and then using the [] operator. Array elements are initialized with default objects at creation time. The elements may be replaced with new objects if so desired.

 

For example, the following would get and set a value on an object array:

 

      Class c("MyClass");

      c.addAttribute("attr", tInteger::instance());

      std::shared_ptr<Object> arr = tObjectArray::value(c, 1);

      std::shared_ptr<Object> myObj = arr->asObjectArray(c)[0];

 

      //replace value

      arr->asObjectArray(c)[0].set("attr", tInteger::value(5));

 

      //replace object

      std::shared_ptr<Object> object = std::make_shared<Object>(c);

      object->set("attr",tInteger::value(5));

      arr->asObjectArray(c)[0] = object;

 

 

D. ClassLoader, ClassStore and JSON Encoded Class Schemas

 

The toolkit provides a mechanism for storing and loading classes from the file system. The class webo.encoding.ClassLoader extends webo.modeling.ClassModel to override the getClass() method so that upon failure to find the class, an attempt is made to load the class from disc.

 

The ClassLoader uses class webo.encoding.ClassStore to store and retrieve a class from disc. Users would extend the interface to provide a different implementation and install it on the ClassLoader, for example, to load classes over the Internet. The following ClassLoader constructor takes a custom ClassStore as an optional parameter:

 

      ClassLoader(std::string rootPath, ClassStore * store = nullptr)

 

Absent the custom parameter, an internal ClassStore would load classes from the disc.

 

The JSON class supports a mechanism for creating and decoding a class schema from/to a Class object. The proprietary schema format is referred to as JavaScript Schema Notation (JSSN), pronounced 'jazzin'. This schema format is independent from proposed standards for JSON Schema, and is used for encoding classes within this toolkit. The following methods are provided on the JSON class (described more fully in a separate section):

 

      std::string stringifySchema(modeling::Class & clazz)

      void parseSchema(std::string & jssnString)

 

The following provides an example of how to use these facilities:

 

ClassModel cm;

Class & c = cm.createClass("MyClass", "this.that.other");

c.addAttribute("attr", tString::instance());

 

JSON json(cm);

std::string schema = json.stringifySchema(c);

ClassStore cs("c:/test");

cs.store("MyClass", schema, "this.that.other");

 

ClassLoader cl("c:/test");

Class & c = cl.getClass("MyClass", "this.that.other");

Object o(c);

 

We start by creating a new class MyClass in the package this.that.other. An attribute is also added to the class.

 

We demonstrate the encoding of the class into JSON and the use of ClassStore to store the encoding to disc. This code is independent of the ClassLoader and allows us to create a test class schema on disc that the ClassLoader can load. First, the JSON class is used to create the schema from the class. Then, an instance of ClassStore is created with a root location of c:/test. It is used to store the schema for MyClass to the file system. This results in the following path and file being created with the schema contents:

 

      c:/test/this/that/other/MyClass

 

Finally, an instance of ClassLoader is created, also with a root location of c:/test. It is used to retrieve the class MyClass from the file system. Given a set of stored class schemas on the file system, this last code block is all that would be needed to retrieve stored classes and begin creating objects from them.

 

 

III.  ENCODINGS

 

A. JSON Encoding

 

Encoding to JSON

 

The toolkit provides a facility for encoding an instance of a user defined class to JSON. The class webo::encoding::JSON provides the desired function. The stringify() method accepts an Object and a corresponding Type as parameters and returns a JSON string. As discussed previously, an Object consists of a set of attributes and attributes values. Those attributes may be objects, arrays or other simple types. Note that different classes containing matching attributes may be used to encode or decode to the same JSON.

 

The JSON object is instantiated using a ClassModel reference that would resolve any of the Types referenced by the inner structure of the object parameter. For example, the following defines an object with various attributes and converts it to JSON:

 

      ClassModel cm;

      Class & c1 = cm.createClass("MyClass");

      Class & c2 = cm.createClass("MyClass2");

      c2.addAttribute("attr1", tInteger::instance());

      c2.addAttribute("attr2", tString::instance());

      c1.addAttribute("attr1", c2);

      c1.addAttribute("attr2", tCharacter::instance());    

 

      std::shared_ptr<Object> obj1 = std::make_shared<Object>(c1);

      std::shared_ptr<Object> obj2 = obj1->get("attr1");

      obj2->set("attr1", tInteger::value(5));

      obj2->set("attr2", tString::value("hello"));

      obj1->set("attr2", tCharacter::value('-'));

 

      JSON json(cm);

      std::string result = json.stringify(obj, c1);

 

The value of result would be:

 

{

   "attr1" :

      {

         "attr1" : 5,

         "attr2" : "hello"

      },

   "attr2" : "-"

}

 

 

Decoding from JSON

 

The toolkit provides a facility for decoding a JSON string into an instance of a user defined class. The class webo::encoding::JSON provides the desired function. The parse() method accepts a std::string containing valid JSON and a corresponding Class as parameters  and returns std::shared_ptr<Object> as an instance of the corresponding user defined class.

 

The JSON object is instantiated using a ClassModel reference that would resolve any of the Types referenced by the inner structure of the JSON string being decoded. For example, continuing the example from above, the following would decode the result string back into an object:

 

      std::shared_ptr<Object> obj3 = json.parse(result, c1);

      assert( obj3->type == c1 );

      assert( obj3->get("attr1")->type == c2 );

      assert( obj3->get("attr1")->get("attr1")->asInteger() == 5 );

      assert( obj3->get("attr1")->get("attr2")->asString() =="hello" );

      assert( obj3->get("attr2")->asCharacter() == '-' );

 

 

B. XML Encoding

 

Encoding to XML

 

The toolkit provides a facility for encoding an instance of a user defined class to XML. The class webo::encoding_xml::XML provides the desired function. The stringify() method accepts an Object and a corresponding Type as parameters and returns a XML string. As discussed previously, an object consists of a set of attributes and attributes values. All basic types are converted to plain text when used as content for an XML Element or as quoted text when used as an XML Attribute value; tBoolean converts to "true" or "false", and tNull converts to "null".

 

A step wise example of how to encode XML from an Object follows.

 

1. Use the following include files and namespaces

 

      #include "webo/encoding_xml/XML.h"

      #include "webo/modeling/Modeling.h"

      #include "webo/modeling/datatype/DataTypes.h"

     

      using namespace webo::modeling;

      using namespace webo::modeling::datatype;

      using namespace webo::encoding_xml;

 

 

2. Define a ClassModel as a holder for dynamically created classes

 

      ClassModel myClasses;

 

3. Define a class for each XML Element

 

      Class & myOuterElement = myClasses.createClass("MyOuterElement");

      Class & myInnerElement = myClasses.createClass("MyInnerElement");

 

4. Define attributes with a name starting with '@' to create simple XML Elements

 

            myOuterElement.addAttribute("@id", tInteger::Instance());

 

5. Define attributes with a name starting with '#' to create XML attributes

 

      myOuterElement.addAttribute("#attr", tString::Instance());

 

6. Assemble the nodes to form the XML structure

 

      myOuterElement.addAttribute("node1", myInnerElmeent);

           

7. Allocate the objects that form the XML structure

 

      Object myOuter(myOuterElement);

      Object myInner(myInnerElement);

      myOuter.set("#attr", "a");

      myOuter.set("@id", tInteger::Value(1));

      myOuter.set("node1", myInner);

 

8. Use the XML utility class to generate the XML data

 

      XML xml(myClasses);

      cout << xml.stringify(myOuter, myOuterElement) << std::endl;

 

 

      <MyOuterElement attr="a">

            <id>1</id>

            <MyInnerElement></MyInnerElement>

      </MyOuterElement>

 

 

Decoding from XML

 

This feature is not currently supported.

 

 

APPENDIX A. WINDOWS INSTALLATION

 

To install the toolkit into your Visual Studio solution, you must first copy the public header files and DLLs into a third party directory to be used by your projects. The public folder of the webo build project contains both. Note that while the Boost libaries are required to build a number of the toolkit's components, a Boost installation is not required to link the built libraries into your application.

 

For example, if your third party directory is defined by the environment variable 3RD_PARTY, you may copy the public folder from a successfully built toolkit into a directory called webo under your third party directory:

 

      <webo solution>/public  --->  $(3RD_PARTY)/webo

 

The following changes would be made to a VS2015 project file to import and use the Webo Toolkit modeling library:

 

      C/C++->General->Additional Include Directories

 

            $(3RD_PARTY)/webo/public

 

      Linker->General->Additional Library Directories

 

            $(3RD_PARTY)/webo/public/lib

 

      Linker->Input->Additional Dependencies

 

            modeling-d.lib (for debug)

            modeling.lib (for release)

 

 

APPENDIX B. LINUX INSTALLATION

 

All libraries are based on standard C++ and are therefore compatible with Linux C++ projects, after building the source code. The file ClassStore.cpp must be modified to use the Korn shell version of the function mkdir(), used to create folders in the file system.
